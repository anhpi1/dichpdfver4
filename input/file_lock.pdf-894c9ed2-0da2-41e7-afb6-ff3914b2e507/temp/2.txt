Pathloss File Locking and Sharing [4]
The June 03, 2002 Pathloss program build introduces pathloss data (pl4) file locking and network (gr4) file sharing when these files are located on a shared network drive. The locking and sharing is activated if the root drive containing the network or pathloss data file is a remote drive. Note that this can fail on a peer to peer network, if a drive is accessed as a local drive. [5]
Pathloss data (pl4) file locking [6]
The locking mechanism, creates a new file with the same name as the Pathloss data files with the extension lck. When the user closes this file, the lock file is erased. If another user attempts to open the file, a prompt is issued that the file is in use and will be opened in read only mode. It is not possible to save the file un- der another name. The program does not check if the lock file has been (deleted) [7]
Network (gr4) file sharing [8]
To enable network (gr4) file sharing, select Files - Shared file operation on the network menu bar. This step must be carried out before loading a network file. To enable network file sharing after a file has been load- ed, select Files - New and then select Files - Shared file operation. Then reload the file. The shared setting is saved in the program option file. [9]
In this mode, a file cannot be saved if the following conditions are not met. [10]
• Each site must have a unique call sign. This is a 15 character identifier and cannot be blank. The same [11]
call sign cannot be used at different sites. The call sign serves as a unique identifier to determine [12]
changes to the sites and links. [13]
• The revision date of the file on disk must be the same as the file in memory. Another users changes can- [14]
not be overwritten. [15]
The sharing mechanism uses a file with the same name as the network file with the extension glk. This file will be referred to a glk file in the following descriptions and the file contains the following information: [16]
• the revision date and change number [17]
• a list of all users who have opened the network file [18]
• a series of records containing the changes made to the file . These records are used to update the indi- [19]
vidual users files [20]
The glk file is created when the first user opens the network file. At this point the file contains the name of the user and the revision date. The change number is set to zero. The program will then begin to monitor the glk file at 30 second intervals to determine if the revsion date has changed. [21]
The glk file is deleted when the last user closes the network file. [22]

Once the network file has been opened by the first user all other users opening the file will be present- ed with a list of users who have opened the file open. The new users are added to the user list and receive revision revision date at the time that they loaded the file. [26]
Suppose a user makes some changes to the file and saves the changes. A change is defined as follows: [27]
• a site has been added [28]
• a site has been deleted [29]
• a site has been moved [30]
• a link has been added [31]
• a link has been deleted [32]
• the Pathloss file associated with the link has been [33]
changed. [34]
When the file is saved, a record of the changes is first created by comparing the file in memory with the file on disk. The glk file will be updated with the new re- vision date, and the record of changes will be append- ed to the file. [35]
In this shared environment, the program checks the glk file every 30 seconds and if a change has oc- curred, the user is notified with the list of changes. Each change can be viewed. [36]
Make changes [37]
The changes are made directly from the glk file. The display will not be reformated. All of the changes in the glk file will be made. [38]
Reload file [39]
The complete file will be reloaded and the display will be reformatted. The reload file option is only avail- able in the network and mapgrid modules [40]
If the cancel button is clicked, the user can update the file by returning to the network or mapgrid module and reloading the file using Files - Open or selecting Files - Shared File Status. The latter produces the same display and options as a change notification in the network or mapgrid modules [41]
Only one change notification is issued even if the user decides to not re-load the file. Once the file has been updated, the change notification will begin again. [42]

Interference Calculations on a shared Network file [46]
The inherent table locking schemes used in the Borland database engine does not allow users to carry out simultaneous interference calculations. An interference calculation creates a set of tables which relate to the network gr4 file. [47]
The June 02, 2002 program build allows the user to specify a private directory for inter- ference calculations. The selection is made at the start of an interference calculation. [48]
Operations such as specifying fade correla- tion will fail if the current network file does not match the database tables in the private directory. [49]
